    A. --------------------------------------------------------------------------------
   SYSTEM DESIGN


  1.   Architecture Design



Edge Layer (Perception):
This layer contains the hardware deployed directly inside the shelter. We use an ESP32 to poll data from the DHT22 and MPU6050 sensors every few seconds. A Raspberry Pi 4 manages the Pi Camera Module V2 to capture visual evidence. To save bandwidth, the camera stays in standby mode and only triggers when the system detects a high-risk event or a technician sends a manual override command.
Network Layer (Transport):
This layer manages the data flow from the devices to the cloud using a 2.4 GHz WiFi connection. The ESP32 publishes sensor telemetry, like a 32.5°C temperature reading, via the MQTT protocol in JSON format. We chose MQTT because it is lightweight and handles real-time monitoring with minimal overhead. For larger files, the Raspberry Pi uploads .jpg images directly to the backend server using HTTPS POST to ensure secure transmission.


Cloud Layer (Application & Processing):
The cloud layer handles all heavy processing and long-term storage. The backend server receives every sensor packet and image file for analysis. Our scikit-learn model processes these inputs to classify the environment into low, medium, or high-risk categories. Simultaneously, the system runs face detection on any incoming images to identify unauthorized personnel.

  2.   Entity Relationship Diagram 


We use a relational database to organize sensor logs and CCTV evidence.
  oo   Shelter and Device Management: 
   The Shelter table acts as the primary entity for all data. This structure allows us to track multiple locations and pinpoint exactly which sensor sent a specific reading.
  oo   Telemetry and Sensor Data Storage: 
   The SensorData table stores temperature and vibration values with a precise timestamp. These logs allow the web dashboard to generate historical graphs so users can spot dangerous patterns early.
  oo   Evidence and Notification Linking: 
   When the system detects a risk, it saves a record in the CCTVEvidence table. We store the image URL and link it to a specific alert ID so technicians can see exactly what triggered the notification.
  oo   Threshold and Risk Configuration: 
   The Threshold table stores critical limits, such as a 45°C temperature ceiling. We can update these safety margins in the database without changing a single line of backend code.

  3.   Machine Learning Logic and Process Design




Data Acquisition and Pre-processing: 
The process kicks off when the ESP32 sends a JSON payload. The system extracts the raw floats and prepares them for the ML model to ensure the classification is accurate
Risk Classification: The model evaluates the vibration magnitude and temperature data alongside the image analysis.
Low Risk: If values remain below the 35°C warning threshold, the system simply logs the data and keeps monitoring.
High Risk: If the model detects a temperature spike or an unknown person, it marks the status as high risk.
Alert Execution: 
During a high-risk event, the system executes two tasks at once.
Evidence Capture: The Raspberry Pi captures a JPEG image and sends it to the server.
Instant Notification: The backend triggers the Telegram API to send a mobile alert to the technician.
System Feedback Loop: 
The system saves the entire incident report to the database before returning to its normal monitoring loop.

  4.   Interface Mock-ups


The Dashboard helps technicians monitor the shelter and respond to incidents quickly.
Web-Based Monitoring Dashboard: 
  oo   Live CCTV Feed: This panel displays the most recent images captured by the Pi Camera. It does not show live video, but rather the specific frames taken during anomalies.
  oo   Real-Time Status Indicators: Gauge charts show real-time temperature and vibration. The color shifts to red if a value passes the safe limit.
  oo   Dynamic Alert Feed: This table logs every recent alert with the time and a link to the evidence
Real-Time Telegram Notification:
  oo   The Telegram bot ensures users stay updated even when they are away from their desks.
  oo   Alert Message: The message includes the specific risk type and the Shelter ID.
  oo   User Interaction and Feedback: Technicians can mark alerts as "Handled" directly through the dashboard to track the response history.

   HIERARCHICAL/ITERATIVE DESIGN

  1.   Software Engineering Design Steps
   We used a top-down software engineering approach to design this monitoring system, ensuring a logical progression from the overarching architecture down to the individual components:
   Architectural Design: We defined the distributed structure of the system, dividing it into the physical Edge Nodes inside the shelter, the Central Processing Server in the cloud, and the User Endpoints for the technicians.
   Interface Design: We established the communication protocols between these blocks. This includes using MQTT for lightweight, real-time telemetry streaming and HTTP REST APIs for handling larger payloads like image uploads.
   Component-Level Design: We mapped out the specific hardware modules, integrated circuits (ICs), and the primitive software functions required to process sensor inputs and execute machine learning classifications.
   
  2.   Hierarchical Block Diagram (Highest to Lowest Level)
The system is broken down into three distinct levels, moving from the macro-architecture down to the discrete hardware and software primitives.
  oo   Level 0 (System Architecture): The highest level grouping the physical Shelter Environment, the Central Processing Server, and the User Endpoints.
  oo   Level 1 (Subsystem Level): This level breaks down the main architecture into functional blocks, including the IoT Sensor Node, Smart CCTV Node, MQTT Broker, Backend API Logic, and Relational Database.
  oo   Level 2 (Lowest Level / Hardware & Primitive Functions): * Discrete Components and ICs: * ESP32 microcontroller acting as the main processing unit for the sensors. * MPU6050 Integrated Circuit (accelerometer and gyroscope) communicating via I2C interface to measure vibration. * DHT22 digital temperature sensor connected via GPIO pins. * Raspberry Pi 4 integrated with a Pi Camera Module V2 for physical evidence capture.
Primitive Software Functions:
  oo   readTemperature(): A software function to fetch digital float values from the DHT22.
  oo   readVibration(): A primitive function to extract acceleration magnitudes from the MPU6050 hardware registers.
  oo   captureEvidence(shelter_id): A hardware-triggering function to activate the camera shutter and save the .jpg file locally before upload.
--------------------------------------------------------------------------------
   


3. Clear Interface Information
To ensure seamless communication between the system blocks, we defined the following interfaces:
  oo   Message Passing: The ESP32 sensor node continuously transmits telemetry data to the MQTT broker using JSON-formatted messages (e.g., {"temp": 34.5, "vib": 0.4, "timestamp": "2026-03-25T20:30:00Z"}). Conversely, the server passes trigger commands to the Raspberry Pi by publishing to a specific MQTT topic (shelter/device_id/camera/trigger).
  oo   Parameter Passing: During risk analysis, the backend logic passes historical telemetry arrays as parameters into the machine learning module using the predictRisk(temp_array, vib_array) function. Additionally, the Raspberry Pi passes image data and authentication tokens via an HTTP POST request payload when uploading evidence to the server.
  oo   Global Variables: Critical system configurations -- such as the database connection URI, MQTT secret keys, Telegram API tokens, and baseline temperature/vibration thresholds -- are stored as global environment variables (.env) on the backend server. This allows all modules to access necessary configurations securely without hardcoding values into the logic.
4. Component References and Libraries Used
  oo   The software implementation relies on the following standard libraries and references to handle hardware interfacing and data processing:
  oo   ESP32 Firmware (C/C++): PubSubClient.h for handling MQTT message passing, Adafruit_Sensor.h and DHT.h for temperature data acquisition, and Adafruit_MPU6050.h to interact with the vibration IC.
  oo   Raspberry Pi (Python): picamera to control the physical camera hardware and requests to manage HTTP REST API interactions.
  oo   Backend & Machine Learning: paho-mqtt for the server-side broker connection, scikit-learn (or TensorFlow Lite) to run the predictive risk classification model, and python-telegram-bot to manage real-time alert dispatches.

   5. System Modelling (UML & Flowcharts)
We utilized standard Unified Modeling Language (UML) and flowcharting techniques to represent both the structural and behavioral aspects of the system.
  oo   Use Case Diagram: Defines the interactions between the Administrator, Technician, AI Module, and the system components. 
   
  oo   Entity Relationship Diagram (ERD): Details the relational database schema, tables, and cardinalities.
   

  oo   Data Flow Diagram (DFD): Maps how telemetry, configuration, and image data move through system processes from the shelter environment to the end-users. To comply with strict structural modeling standards, the DFD is hierarchically decomposed into multiple levels: starting from a macro-level Context Diagram (Level 0), breaking down into main sub-processes (Level 1), detailing the anomaly detection logic (Level 2), and reaching the deepest algorithmic flow of the Machine Learning classification (Level 3).
